## 오늘의 결정

### Pixel Art Crawler — 왜 만들기로 했는가

2일차까지 Addressable 인프라, VFX, UI 테마를 갖춰 놨지만, 실제로 채워 넣을 **아트 에셋이 105개나 부족**했다. 하나하나 수작업으로 찾아 다운로드하고 Unity에 맞게 설정하는 건 시간이 너무 많이 든다. AI로 프로토타입을 빠르게 돌리려면, 아트 수집부터 Unity 연동까지 **한 번에 처리되는 파이프라인**이 필요했다.

그래서 **이미지 크롤러**를 제작하기로 했다. OpenGameArt, itch.io, Spriters Resource에서 검색 → 다운로드 → `.meta` 자동 생성까지 묶어서 CLI 한 줄로 실행할 수 있게 만드는 게 목표다.

### 포트폴리오 용도라는 전제

이번 프로젝트는 **포트폴리오 용도**이며, 실제 상업적 출시는 계획에 없다. 따라서 에셋 라이선스를 상업적 관점에서 굳이 따지지 않아도 된다. 레퍼런스·학습·포트폴리오로 제한해서 사용하는 선에서만 운영할 예정이다.

## 프로젝트 전환 — 유튜브 영상의 영향

[이 유튜브 영상](https://www.youtube.com/watch?v=7vihh_G_434&feature=youtu.be)을 시청한 후, **전체 프로젝트를 다시 만들기로** 결정했다. 기존 방향에서 전환하여 새로 출발하는 계기가 됐다.

### 영상 요약 — AI를 길들이는 네 가지 시스템

6개월간 Claude Code를 활용해 책 300~400권 분량의 프로그램을 완성한 8년 차 개발자가 공개한 내용이다. 혼자서 몇 개월 안에 규모 있는 프로그램을 완성한 비결은 **AI를 그냥 쓰는 게 아니라, AI가 잘 일하도록 시스템을 만든 것**이었다.

#### AI의 문제점 — 기억력이 부족한 신입

AI는 처음엔 지시를 잘 따르다가, 어느 순간 다른 일을 하거나 지시 내용을 잊어버린다. 옆에 매뉴얼이 있어도 보지 않고 진행해서 같은 실수를 반복하기도 한다. 영상에서는 이런 AI를 **"30분마다 자신이 하던 일을 잊어버려 계속 지시를 다시 해줘야 하는 신입 직원"**에 비유했다. 능력은 뛰어나지만, 기억력과 자기 점검이 약하다는 뜻이다.

#### 네 가지 시스템 개요

| 시스템 | 설명 | 비유 |
|--------|------|------|
| **자동 매뉴얼** | AI가 작업 전에 관련 매뉴얼을 자동으로 확인하게 함. 후크로 지시 전·후에 매뉴얼·체크를 강제 | 선배가 신입에게 "매뉴얼 참고해" 귀띔 |
| **작업 기억** | 계획서, 맥락 노트, 체크리스트를 외부 기억으로 활용. 대화가 길어져도 목표 유지 | 금붕어 기억력 극복용 메모장 |
| **자동 품질 검사** | 수정 파일 기록 → 오류 자동 체크 → 셀프 체크 리마인더 → 수정 유도 | 공장 컨베이어 벨트 끝의 검수대 |
| **전문 에이전트** | 품질관리, 테스트, 기획 등 역할별 AI를 두고 협업. 결과물 보고서·코드 리뷰까지 요청 | 역할 분담된 AI 팀 구성 |

#### 자동 매뉴얼 — 핵심은 강제 적용

가이드라인을 아무리 길게 써도 AI가 안 읽으면 의미가 없다. 그래서 **시작 전 알림**(지시 읽기 전에 관련 매뉴얼 확인 유도)과 **완료 후 체크**(결과 검토 후 누락·오류 질문) 두 가지 후크로, 매뉴얼을 **꼭 보게** 만든다. 매뉴얼은 목차·챕터·파일로 나눠서 필요한 부분만 로드하게 하면 토큰도 40~60% 절약된다고 한다.

#### Claude의 Hook 기능 — 매뉴얼 강제 적용의 시동 장치

Hook은 **AI가 지시를 읽기 전·후에 특정 동작을 끼워 넣는 기능**이다. 매뉴얼을 그냥 옆에 두면 AI가 무시하니까, Hook으로 "이 시점에 반드시 매뉴얼을 확인하라"는 흐름을 강제로 넣는 것이다.

영상에서 활용한 예시는 다음과 같다. 지시 **읽기 전**에는 관련 매뉴얼을 꼭 확인하게 하고(시작 전 알림), 작업 **끝난 뒤**에는 스스로 점검하도록 유도한다(완료 후 체크). 가이드라인만 작성해 두는 것이 아니라, Hook으로 이 시점에서 반드시 매뉴얼이 사용되게 만드는 게 핵심이다.

##### Claude Code가 제공하는 Hook 이벤트

Claude Code는 워크플로우의 다양한 지점에서 실행되는 여러 훅 이벤트를 제공한다. 각 이벤트는 서로 다른 데이터를 수신하며, Claude의 동작을 세밀하게 제어할 수 있다.

| Hook 이벤트 | 실행 시점 | 특징 |
|-------------|-----------|------|
| **UserPromptSubmit** | 사용자가 프롬프트를 제출한 직후, Claude가 처리하기 **전** | 매뉴얼 확인 유도·시작 전 알림에 활용. 영상의 "시작 전 알림"에 해당 |
| **PreToolUse** | 도구 호출 **전** | 차단 가능 — 특정 도구 사용 전에 검사·승인·매뉴얼 주입 |
| **PostToolUse** | 도구 호출 **완료 후** | 결과 검토·셀프 체크 리마인더 유도. 영상의 "완료 후 체크"에 활용 |
| **Stop** | Claude가 응답을 **마칠 때** | 작업 종료 직전 점검·누락 확인 |
| **SubagentStop** | 서브에이전트 작업이 **완료될 때** | 하위 에이전트 결과 검수 |
| **PreCompact** | Claude Code가 **컴팩트 작업을 실행하려 할 때** | 컨텍스트 압축 전 인터셉트 |
| **PermissionRequest** | 권한 대화상자가 **표시될 때** | 허용·거부 제어 가능 |
| **Notification** | Claude Code가 **알림을 보낼 때** | 알림 내용·타이밍 제어 |
| **SessionStart** | 새 세션 시작 또는 기존 세션 **재개 시** | 세션 초기화·환경 설정 주입 |
| **SessionEnd** | 세션 **종료 시** | 정리·로그·후속 작업 트리거 |

이 중 **UserPromptSubmit**(시작 전)과 **Stop** 또는 **PostToolUse**(완료 후)를 조합하면, 영상에서 말한 "시작 전 알림 + 완료 후 체크" 흐름을 구현할 수 있다. 필요에 따라 PreToolUse로 도구 사용 전 매뉴얼을 강제하거나, SessionStart로 세션마다 공통 가이드를 주입하는 식으로 확장할 수 있다.

#### 작업 기억 — 계획서·맥락 노트·체크리스트

한 번에 여러 작업을 시키지 말고, 한두 개씩 시키고 결과 확인 후 다음으로 넘긴다. 계획이 승인되는 순간 바로 멈추고, 세 가지 문서(계획서, 맥락 노트, 할 일 체크리스트)로 저장한 뒤 이어서 진행하도록 하면, 30분 전 계획이 두 시간 뒤에도 유지된다.

#### 자동 품질 검사 — 검수 라인 구축

AI가 "다 했습니다"라고 해도 실제로는 실수가 많다. 그래서 **수정 기록 → 완료 후 오류 검사 → 수정 요청 or 전문 수리 담당 추천** 흐름을 자동화한다. 공장에서 원자재 스펙 확인, 생산 라인 검수, 불량 시 재처리하는 것처럼, 끝까지 관리하는 게 포인트다.

**핵심 메시지:** AI는 그냥 쓰면 50점, 시스템을 만들어 주면 95점이 된다. ChatGPT·Gemini 등 어떤 도구든 마찬가지다. 도구 탓 전에 **하니스(Harness) 시스템 환경**을 갖추는 게 우선이다.

## 프로젝트에 적용할 점 — 영상 인사이트

이번 리메이크부터는 영상에서 배운 방식을 적용해보려 한다.

- **자동 매뉴얼**: Cursor Rule이나 Skill로 프로젝트 네이밍·폴더 구조·코딩 컨벤션을 고정해 두고, 작업 전에 반드시 읽히게 설정
- **작업 기억**: `AGENTS.md`나 `CONTEXT.md`에 계획서·맥락 노트·체크리스트를 두고, 대화가 길어져도 컨텍스트가 유지되게 유지
- **자동 품질 검사**: 린트·포맷터·테스트를 커밋 전/CI에 걸어서, AI가 수정한 코드도 자동으로 점검
- **전문 에이전트**: 코드 리뷰·테스트·문서화를 각각 다른 에이전트(또는 프롬프트)에게 맡기는 방식 시도

크롤러 개발부터 이런 식으로 시스템을 같이 쌓아 가면, 나중에 리메이크 프로젝트도 더 일관되게 진행할 수 있을 것 같다.
